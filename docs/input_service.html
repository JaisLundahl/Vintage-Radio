Input Service Technical Deep Dive

This service manages user interaction through physical buttons, rotary encoders, and ADC-based mode selection.

1. Event Architecture & Queue

The system uses a producer-consumer model with a small internal ring buffer (queue) to ensure no user inputs are missed during busy processing cycles.

  • The Event Structure: Every InputEvent contains the event type (press, rotation, etc.), the AppMode active when the event occurred, and a timestamp.
  • qpush() / qpop(): These helper functions manage a circular buffer of size 8. If the queue overflows, the oldest event is discarded to make room for the newest interaction.

2. Advanced Button Handling

updateButton(BtnState& b)

This function implements a sophisticated state machine for a single button.

  • Software Debounce: Ignores state changes faster than 20ms (DEBOUNCE_MS) to filter out mechanical switch noise.
  • Multi-stage Timing:
    TypeThresholdEvent Triggered
    Short Press< 800msOn Release
    Long Press800msDuring Hold
    Very Long Press5000msDuring Hold
  • Release Logic: If a "Very Long Press" was triggered, the system can optionally fire a EV_VERY_LONG_RELEASE event.

3. Rotation Decoding

updateEncoderRotary()

When configured in INPUT_MODE_ROTARY, this function decodes Quadrature signals from pins A and B.

  • Direction Table: Uses a lookup table (dirTable) based on the previous and current state of both pins to determine CW (Clockwise) or CCW (Counter-Clockwise) movement.
  • Velocity Limiting: Includes a debounce check to prevent "ghost" rotations if the encoder is turned too fast or has mechanical bounce.

4. Mode Selection via ADC

decodeModeFromAdc(uint16_t raw)

Instead of a digital switch, the system uses an analog voltage divider to determine the current application mode.

  • Voltage Windows:
    • BT: < 400 (~0V)
    • MP3: 1400 - 2600 (~1.65V)
    • NET: > 3000 (~3.3V)
  • Hysteresis & Debounce: updateModeSelector() requires the voltage to stay within a window for 200ms (MODE_DEBOUNCE_MS) before the EV_MODE_CHANGED event is fired.

5. Public Interface

  • inputsInit(): Configures pins as INPUT_PULLUP and performs the first read to establish the baseline state.
  • inputsTick(): Should be called frequently (e.g., in the main loop or a timer task) to process physical signals into events.
  • inputsGetEvent(InputEvent& out): The primary way for the application to consume user inputs. Returns true if an event was retrieved from the queue.
