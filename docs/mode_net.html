Network Radio Mode Technical Deep Dive

This module implements an Internet Radio client supporting MP3 and AAC streams, PLS/M3U playlist parsing, and robust Wi-Fi connection management.

1. Wi-Fi Management & Provisioning

The system relies on /system/netradio/WIFI.txt on the SD card to manage credentials.

  • Smart Scanning: connectWiFiFromFile() scans the local airwaves and compares found SSIDs against the stored list, automatically selecting the access point with the strongest RSSI.
  • Event Handlers: Uses onWiFiEvent to detect disconnections. If the signal is lost, it signals the audio task to stop and sets a s_needResumePlay flag to auto-resume once the IP is regained.

2. Stream Acquisition & Parsing

playUrl(const String& url)

This is the core network engine. It handles the complexities of modern web streaming:

  • Protocol Handling: Automatically switches between WiFiClient and WiFiClientSecure (Insecure mode) based on whether the URL is HTTP or HTTPS.
  • Playlist Parsing: Recursively handles .m3u, .m3u8, and .pls files. It extracts the raw stream URL from these text-based formats.
  • Manual Redirects: Because standard HTTP clients often struggle with switching from HTTP to HTTPS during a 301/302 redirect, this module handles redirects manually to ensure the correct secure client is initialized.
Memory Management: Uses lazy-initialization for MP3DecoderHelix and AACDecoderHelix to conserve heap memory until the Content-Type of the stream is confirmed.

3. The Network Audio Task

A FreeRTOS task (netTask) runs at a high frequency (2ms) to prevent audio buffer underflows (stuttering).

  • Data Pumping: Calls s_copy->copy() which pulls data from the HTTP stream, decodes it via Helix, and pushes it to the AudioInfoGuard in the AudioService.
  • Stream Watchdog: Monitors s_lastAudioData. If no data is received for 10 seconds (kStreamStallMs), it assumes the stream is dead and initiates a retry.
  • Command Queue: Uses volatile flags (s_cmdStop, s_cmdPlay) to allow the main thread to control the task safely without mutex-induced deadlocks.

4. Tuning Experience

Similar to the MP3 mode, the Network mode uses a "Wait Tone" to mask the latency of Wi-Fi connections and buffer filling.

  • Wait Tone Ownership: When tuning, mode_net releases ownership of the I2S hardware and plays wait.mp3 locally. This ensures the user hears immediate feedback while the ESP32 is busy performing DNS lookups or HTTP handshakes.
  • Station Loading: Loads up to 16 stations from /system/netradio/stations.txt, parsed as Name|URL|Type.

5. Performance Monitoring

The service logs download performance to the Serial console every second (kRxPrintIntervalMs). This helps in diagnosing whether audio stuttering is due to slow Wi-Fi or server-side throttling.
